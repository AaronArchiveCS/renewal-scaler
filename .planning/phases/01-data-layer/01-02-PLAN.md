**### Step 3: Parallel Data Fetch**

Add instructions for Claude to fetch data from PostHog and Stripe in parallel. Use the shop ID list from Step 2.

**PostHog — UGC Usage (per D-06, DATA-01, DATA-06):**

Instruct Claude to use `mcp__claude_ai_PostHog__query-run` with the following HogQL query. For multi-workspace customers, use `IN` with all shop IDs from Step 2:

```sql
SELECT
  toStartOfMonth(timestamp) AS month,
  COUNT(*) AS ugc_count
FROM events
WHERE event = 'crm.shop_item.created'
  AND properties.shop_id IN ('{shop_id_1}', '{shop_id_2}', ...)
  AND timestamp >= now() - INTERVAL 3 MONTH
GROUP BY month
ORDER BY month ASC
LIMIT 10
```

Include this note: "Do NOT set filterTestAccounts to true — it returns empty results in this PostHog project."

From the results, calculate:
- **3-month rolling average** = sum of all 3 months / 3 (if fewer than 3 months have data, divide by the number of months with data)
- **Trend direction** per D-12: Compare month1->month2 and month2->month3. If both increasing: "Growing". If both decreasing: "Declining". Otherwise: "Stable".
- If 0 UGC events across all months, note: "0 UGC created in past 3 months — verify customer is active." Do NOT treat zero as missing data (it may be a valid low-activity customer).

**PostHog — Active Seats (per D-07, DATA-04):**

```sql
SELECT COUNT(DISTINCT person_id) AS active_seats
FROM events
WHERE event = '$pageview'
  AND properties.shop_id IN ('{shop_id_1}', '{shop_id_2}', ...)
  AND timestamp >= now() - INTERVAL 90 DAY
```

Include this note: "Use $pageview as the activity proxy, NOT auth.user.logged_in (that event only started March 24, 2026 — not enough historical data)."

**Stripe — Plan, Pricing, and UGC Limit (per D-05, D-08, DATA-02, DATA-03):**

The Stripe lookup strategy depends on what was discovered in Plan 01-01 about linking fields. Write the skill instructions to try both approaches:

1. **Primary approach:** Check if the HubSpot company record has a `stripe_customer_id` property. If populated, use:
   ```
   mcp__claude_ai_Stripe__list_subscriptions
     customer: "<stripe_customer_id>"
   ```

2. **Fallback approach:** If no `stripe_customer_id`, search Stripe by shop_id metadata:
   ```
   mcp__claude_ai_Stripe__search_stripe_resources
     resource: "subscriptions"
     query: "metadata['shop_id']:'<shop_id>'"
   ```

3. **Last resort:** Search Stripe customers by company name:
   ```
   mcp__claude_ai_Stripe__search_stripe_resources
     resource: "customers"
     query: "name:'<company_name>'"
   ```
   Then use the customer ID to list subscriptions.

Once a subscription is found, extract:
- **Plan name**: From the product object linked to the subscription's price. Use `mcp__claude_ai_Stripe__fetch_stripe_resources` with `resource: "products/<product_id>"` to get the product name.
- **Monthly price**: From the price object. If `recurring.interval` is "month", use `unit_amount / 100`. If "year", divide by 12.
- **Annual price**: If monthly, multiply by 12. If annual interval, use `unit_amount / 100` directly.
- **UGC limit**: Search the subscription metadata, price metadata, and product metadata for any field containing "ugc", "limit", "cap", or "credit". The exact field name is unknown — the skill must instruct Claude to inspect the full subscription JSON and identify the UGC limit field. Document what was found for future reference.

Include this note: "If the UGC limit field cannot be found in Stripe, surface this as a warning: 'UGC limit not found in Stripe subscription metadata. Check subscription [sub_id] manually.' Do NOT hard-stop — classify as secondary per D-13 exception for first-time discovery."
